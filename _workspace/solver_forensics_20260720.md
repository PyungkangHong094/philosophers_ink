# 레벨 랩 솔버 위양성 포렌식 (2026-07-20)

조사자: solver-forensics · 대상: `tool/level_lab/out/` 아카이브 21레벨 위양성

## 결론 (TL;DR)

**솔버·롤아웃·리셋 코드에는 결함이 없다.** 현재 솔버는 결정적이고 건전하다(main==isolate,
run-to-run 동일, 재해결 시 진짜 클리어되는 해를 찾거나 UNSOLVED를 정직하게 보고).

위양성의 근본 원인은 **`out/` 아카이브가 현재 코드·레벨로 만든 산출물이 아니라는 것** —
즉 **스테일(오래된) 데이터**다. 아카이브에 박힌 21개 해는 현재 물리·레벨에서 재생하면 클리어되지
않으며(0/44~51/60 정체), 같은 레벨을 지금 재해결하면 **전혀 다른, 실제로 클리어되는 해**가 나온다.

**영향 범위는 좁다.** 아카이브를 소비하는 커밋된 도구는 `bake_hints.dart` 하나뿐이고, 이 도구는
쓰기 전 원본 해를 fresh 세션으로 재검증하므로 21개를 전부 `null(verify-fail)`로 **정확히 걸러냈다**.
따라서 **거짓 힌트는 구워지지 않았다.** 별점·optimal_ink·치즈 봉쇄는 아카이브를 자동 소비하는
파이프라인이 없어 직접 오염은 없으나, **사람/에이전트가 이 아카이브의 min_ink·solvable을
신뢰해 내린 판단은 무효**다(특히 아직 실행 안 된 "솔버 최소해로 optimal_ink 승격" 계획).

## 반증한 가설 (증거)

리셋 완전성·세션 재사용 누수·config 차이·물리 코드 변경 — **전부 원인이 아니다.**

| 검증 | 방법 | 결과 |
|---|---|---|
| 리셋 완전성 | fresh 세션 reset 직후 grid.hash vs 1400틱 굴린 뒤 reset 해시 | **동일** (h0==h1) — 그리드 잔류 없음 |
| 세션 재사용 누수 | 재사용 1세션에서 동일 후보 2회·20회 연속 롤아웃 | **매번 동일**, 최초 solved 회차 = −1(한 번도 아님) |
| config 의존 | 아카이브 best를 sweep-cfg/bake-cfg/forbidT·F 4종으로 재생 | **전부 solved=false** (config 무관) |
| 결정성 | solveLevel main 3회 + Isolate.run 비교 | **완전 동일** (min_ink·rollouts·best 일치) |
| 물리 코드 변경 | git: sim/gameplay/core 커밋일 vs 09:14 스윕 | 물리 최종 커밋 = 07-19 17:35·07-20 00:08, **스윕보다 앞섬** (동일 물리) |
| 레벨 변경 | `git diff HEAD assets/levels/level_016.json` | **비어 있음** (07-18 커밋본 그대로, byte-identical) |

`GameState.reset()`/`Rules.reset()`/`FlaskSystem.reset()`/`Emitter.resetRuntime()`/`InkBudget.reset()`
는 모두 완전하다(grid·rng 시드·중력·move-stamp·스캔방향·플라스크 카운트·오염·잉크·잔량·타임아웃 복원).
기믹(gate/portal/temp-zone)은 불변 데이터라 누수원이 아니다.

## 근본 원인 (재현 증거)

### 1. 아카이브 해가 재생되지 않는다 (L016 픽스처)
- 아카이브 best: `chalk(91,68)->(15,62)` solved=true **ticks=1014** min_ink=162.
- fresh 세션 재생(모든 config): **solved=false, 최대 10/44 정체.**
- **현재 솔버 재해결**(동일 seed/config): `chalk(93,67)->(17,62)` min_ink=161 → fresh 재생 **CLEAR 44/44 @847틱.**
- 좌표가 아카이브 대비 **일관되게 +2 x-shift** — 아카이브가 미묘하게 다른 지오메트리를 푼 흔적.

### 2. 현재 솔버는 건전하다 (재현 안 됨)
| 레벨 | 아카이브(스테일) | 현재 솔버 재해결 | fresh 검증 |
|---|---|---|---|
| L016 | (91,68) ink162 t1014 | (93,67) ink161 t847 | **CLEAR 44/44** |
| L021 | ink**4** t428 (치즈) | frost(95,72) ink**183** t582 | **CLEAR 28/28** |
| L045 | ink**4** t710 (치즈) | **UNSOLVED** (2700 롤아웃) | — (위양성 재현 안 함) |

현재 솔버는 아카이브의 ink=4 치즈 위양성을 **한 번도 재현하지 못한다** — 진짜 해를 찾거나 정직하게 미해결 보고.

### 3. 아카이브는 라이브 스윕 산출물이 아니다 (mtime 물증)
- `out/` **77개 파일 전부** 09:14:50.966 ~ 09:14:51.019 **약 250ms 버스트**로 기록됨.
- 그러나 각 json의 self-report `elapsed_ms`는 레벨당 ~250초(L016=258,886ms). concurrency 8로
  77레벨이면 실제 벽시계 ~40분. `sweep.dart`는 워커가 레벨을 끝낼 때마다 **즉시** `writeResultJson`을
  호출하므로(증분 기록) — 진짜 라이브 스윕이면 mtime이 ~40분에 걸쳐 흩어져야 한다.
- **250ms 버스트 = 증분 기록이 아님.** 미리 계산된 결과가 벌크로 **복사/복원**된 흔적.
  같은 버스트에서 `assets/levels/`도 재기록됨(L016 asset 09:14:50.708, git-clean=내용 동일).
- 즉 **09:14의 "재스윕"은 실제 재계산이 아니라 오래된 아카이브의 벌크 복원**이었고,
  fresh해 보이는 mtime이 스테일 데이터를 최신처럼 위장했다. 내부 데이터는 현재 물리
  (07-19 "가장자리 배수" 개정 + 07-20 "제한 시간" 도입) **이전 세대**의 것이다 —
  당시엔 입자 더미가 배수 없이 쌓여 플라스크에 닿았기에 ink=4 치즈·다른 지오메트리 해가 유효했다.

## 수정 방안 (솔버 코드 결함 아님 → 계획 보고)

1. **재스윕 (필수)**: 깨끗·정적 워킹트리에서 ch2~4 재스윕. **현 워킹트리는 레벨 10개가
   uncommitted-modified**(012·013·014·015·017·019·025·034·051·064) — 스윕 중 레벨 파일이
   또 변하면 같은 오염 재발. 재스윕은 커밋된 정적 상태에서만.
2. **verify-on-write 가드 (권장, 재발 방지)**: `sweep.dart`가 아카이브에 해를 쓰기 전,
   `bake_hints`가 하듯 fresh 세션 롤아웃으로 실제 클리어를 재검증하고 실패하면 그 해를
   기록하지 않는다. 이러면 스테일·위양성 해가 아카이브에 **물리적으로 남을 수 없다.**
3. **아카이브 프로비넌스 스탬프 (권장)**: 결과 json에 코드 git SHA + 레벨 파일 해시를 박아,
   소비 시점에 현재와 불일치하면 stale로 거부. mtime은 신뢰 불가임이 이번에 증명됨.
4. **치즈 프로브 상한 정합 (2차 하드닝)**: 빈-후보 프로브 `probeCfg.tickCap≥2400`이 레벨 실제
   제한시간(`timeLimitTicks`)을 초과할 수 있어, 게임에선 타임아웃될 "치즈"를 solvable로 볼
   여지가 있다. 프로브/탐색 tickCap을 레벨 제한시간에 클램프 권장(이번 21건의 직접 원인은 아님).
5. **star_thresholds 재저작**: L067·068·069의 2/1 퇴화값은 재스윕 결과와 무관하게 손봐야 함.
6. **assets/levels의 베이크 결과(hint_stroke 11개)는 건드리지 말 것** — 정상 검증 통과분.
